# المرحلة 35 — التوسع وما بعد الإطلاق (Scaling & Growth)

## 🧠 تطوير تنفيذي إضافي — Post-Launch Scale Control Pack

النمو بعد الإطلاق لازم يكون بدون rewrite وبدون paywall. كل scaling decision يتحول لـ threshold + ADR.

### Scale Threshold ADR Trigger

```tsx
export const SCALE_THRESHOLDS = {
  dbCpuPct: 70,
  dbConnectionsPct: 75,
  p95QueryMs: 80,
  replicaLagSeconds: 10,
  monthlyStorageGrowthPct: 25,
  costPerActiveUserUsd: 0.25,
} as const

export function evaluateScaleADR(m: ScaleMetrics) {
  const reasons: string[] = []
  if (m.dbCpuPct > SCALE_THRESHOLDS.dbCpuPct) reasons.push('db_cpu')
  if (m.dbConnectionsPct > SCALE_THRESHOLDS.dbConnectionsPct) reasons.push('db_connections')
  if (m.p95QueryMs > SCALE_THRESHOLDS.p95QueryMs) reasons.push('query_latency')
  if (m.replicaLagSeconds > SCALE_THRESHOLDS.replicaLagSeconds) reasons.push('replica_lag')
  return reasons.length ? { required: true, reasons } : { required: false, reasons: [] }
}
```

### Growth Ethics

- Referral rewards cosmetics فقط.
- لا leaderboards ولا مقارنة اجتماعية.
- لا paywall عند الضغط على الموارد؛ استخدم rate limits عادلة لحماية الخدمة.
- Cost dashboards للتشغيل، مش لتقييد المستخدمين.

## 🧠 Master Architect Upgrade — Scale Without Rewrite

- **Scale thesis:** قبل الشاردينج، استنفد indexing, query plans, read models, caches, pgBouncer, replicas, archival. sharding قرار ADR بعد thresholds.
- **Tenant-aware scaling:** أي archival/read replica/shard routing يحافظ على `workspace_id` isolation، auditability، وconsistent permission checks.
- **Cost as SLO:** cost per active user, AI cost per request, storage growth، وegress يتم قياسهم وتنبيههم؛ النمو لا يفتح paywall.
- **Backpressure:** workers/webhooks/AI/import/search indexers يحتاجوا queues, rate limits, circuit breakers، وload shedding graceful.
- **Growth ethics:** referral/community/roadmap لا يستخدموا dark patterns ولا social comparison. rewards cosmetics فقط ولا تفتح ميزات مغلقة.

## 🎯 نظرة عامة

بعد launch ناجح، دي المرحلة اللي بتجهّز النظام لـ 100x growth بدون عمل rewrite. بنفصل الـ hot data عن cold data بـ archival، بنعمل sharding strategy للـ user data، بنفصل read وwrite workloads على replicas، بنبني CDN multi-tier للـ assets، بنحسّن الـ background workers لـ horizontal scale، وبنبني observability عميق (tracing + profiling + APM) عشان نلاقي الـ bottlenecks قبل ما توصل للمستخدم. الهدف: النظام يتحمل مليون مستخدم بدون تغييرات جوهرية.

### 1) Data archival pipeline

```sql
-- supabase/migrations/0070_archival.sql
BEGIN;
-- Archive table mirror بدون indexes ثقيلة
CREATE TABLE public.tasks_archive (LIKE public.tasks INCLUDING DEFAULTS);
CREATE INDEX tasks_archive_user_idx ON public.tasks_archive(user_id);

CREATE OR REPLACE FUNCTION public.archive_old_tasks() RETURNS INT
LANGUAGE plpgsql SECURITY DEFINER
SET search_path = public, pg_temp AS $$
DECLARE moved INT;
BEGIN
  WITH old_tasks AS (
    DELETE FROM tasks
    WHERE completed_at \< now() - interval '1 year'
      AND is_archived = TRUE
    RETURNING *
  )
  INSERT INTO tasks_archive SELECT * FROM old_tasks;
  GET DIAGNOSTICS moved = ROW_COUNT;
  RETURN moved;
END $$;
COMMIT;
```

// src/lib/db/sharding.ts
import { createHash } from 'crypto'

const SHARDS = parseInt(process.env.DB_SHARDS ?? '1', 10)

export function shardForUser(userId: string): number {
  const hash = createHash('sha256').update(userId).digest('hex').slice(0, 8)
  return parseInt(hash, 16) % SHARDS
}

export function getShardConnection(userId: string) {
  const idx = shardForUser(userId)
  return clients[idx]
}

### 8) Capacity planning

```tsx
// scripts/capacity-plan.ts
interface Metric {
  current: number
  monthlyGrowth: number  // 0.15 = 15%
  capacity: number
  buffer: number          // 0.5 = aim to keep at 50%
}

const metrics: Record<string, Metric> = {
  dau:           { current: 50_000,  monthlyGrowth: 0.20, capacity: 500_000,  buffer: 0.5 },
  db_connections:{ current: 200,     monthlyGrowth: 0.18, capacity: 1000,     buffer: 0.6 },
  redis_memory_gb:{current: 5,       monthlyGrowth: 0.20, capacity: 50,       buffer: 0.7 },
  s3_storage_tb: { current: 1,       monthlyGrowth: 0.25, capacity: 100,      buffer: 0.5 },
  ai_tokens_million:{current:50,     monthlyGrowth: 0.30, capacity: 1000,     buffer: 0.5 },
}

for (const [name, m] of Object.entries(metrics)) {
  const monthsUntilFull = Math.log(m.capacity * m.buffer / m.current) / Math.log(1 + m.monthlyGrowth)
  console.log(`${name}: ${monthsUntilFull.toFixed(1)} months until ${(m.buffer * 100)}% capacity`)
}
```

## 🧠 توسعة تنفيذية ضخمة — AI-Ready Notification Delivery Operating System

هذه المرحلة لا تُكتب كـ “إرسال إشعارات” فقط؛ لازم تتبني كـ **نظام إيصال كامل Delivery Operating System** يفهم نية الحدث، يختار القناة الصح، يحترم موافقة المستخدم، يراعي quiet hours، ويشرح للـ AI/Claude Code بالضبط أين يكتب الكود، ما هي الجداول، ما هي الـ contracts، وما هي اختبارات القبول.

<aside>
🤖

تعليمات للـ AI المنفذ: لا تبدأ من UI. ابدأ من Domain Events → Notification Intent → Preference Resolution → Delivery Queue → Provider Adapter → Audit/Observability. أي إرسال مباشر من route إلى provider ممنوع.

</aside>

### الهدف الهندسي النهائي

### 1. Database Schema (The Bedrock)

#### Migration 0024 — `notifications_system`

```sql
-- 1. Notification Types (Static Registry)
-- Examples: 'task_due', 'workspace_invite', 'system_alert', 'habit_reminder'
CREATE TABLE notification_types (
  id TEXT PRIMARY KEY, -- 'TASK_DUE'
  priority TEXT CHECK (priority IN ('URGENT', 'NORMAL', 'SILENT')) DEFAULT 'NORMAL',
  default_channels TEXT[] DEFAULT '{"in_app"}', -- in_app, email, push
  can_unsubscribe BOOLEAN DEFAULT TRUE
);

-- 2. User Notification Preferences
CREATE TABLE user_notification_settings (
  user_id UUID REFERENCES profiles(id) ON DELETE CASCADE,
  notification_type_id TEXT REFERENCES notification_types(id) ON DELETE CASCADE,
  channels TEXT[] NOT NULL, -- e.g., '{"in_app", "email"}'
  is_enabled BOOLEAN DEFAULT TRUE,
  PRIMARY KEY (user_id, notification_type_id)
);

-- 3. The Central Notification Ledger (Audit & History)
CREATE TABLE notifications (
  id TEXT PRIMARY KEY DEFAULT 'ntf_' || lower(hex(random_bytes(10))),
  user_id UUID REFERENCES profiles(id) ON DELETE CASCADE,
  workspace_id TEXT REFERENCES workspaces(id) ON DELETE CASCADE,
  
  type_id TEXT REFERENCES notification_types(id),
  title TEXT NOT NULL,
  body TEXT NOT NULL,
  data JSONB DEFAULT '{}', -- Custom data like { "task_id": "...", "link": "/p/..." }
  
  read_at TIMESTAMPTZ,
  archived_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT now()
);

-- 4. Delivery Queue (For Background Jobs - BullMQ)
CREATE TABLE notification_delivery_log (
  id TEXT PRIMARY KEY DEFAULT 'ndl_' || lower(hex(random_bytes(10))),
  notification_id TEXT REFERENCES notifications(id) ON DELETE CASCADE,
  channel TEXT NOT NULL, -- 'email', 'push'
  provider_id TEXT, -- 'resend_123'
  status TEXT DEFAULT 'pending', -- pending, sent, failed
  error_message TEXT,
  sent_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT now()
);
```

### 2. Domain logic: The Dispatcher (L5 - Core lib)

#### `lib/notifications/dispatcher.ts`

```typescript
export async function dispatchNotification(args: {
  userId: string;
  typeId: string;
  title: string;
  body: string;
  data?: Record<string, any>;
  workspaceId: string;
}) {
  // 1. Resolve User Preferences
  const settings = await db.notificationSettings.find(args.userId, args.typeId);
  if (!settings?.is_enabled) return;

  // 2. Create in-app record (Primary Record)
  const ntf = await db.notifications.create({ ...args });

  // 3. Queue external deliveries based on preferences
  for (const channel of settings.channels) {
    if (channel === 'in_app') continue;
    
    await notificationQueue.add('deliver', {
      notificationId: ntf.id,
      channel,
      userId: args.userId
    }, { 
      priority: ntf.priority === 'URGENT' ? 1 : 10,
      attempts: 3
    });
  }
}
```

### 3. Frontend Implementation Rules (Aesthetics & UX)

1.  **Notification Center UI:**
    - يجب أن يكون Drawer جانبي أو Popover عند الجرس.
    - يدعم "Mark all as read".
    - تصفية حسب "Unread" فقط.
    - عرض Empty State مبهر (Zenith Illustrations).
2.  **Toasts (Realtime):**
    - استخدام `sonner` أو `react-hot-toast`.
    - عند وصول إشعار و المستخدم فاتح التطبيق، يظهر Toast فوراً عبر Supabase Realtime subscription على جدول `notifications`.
3.  **Quiet Hours (UX Detail):**
    - إضافة حقل `quiet_hours_start` و `quiet_hours_end` في إعدادات المستخدم.
    - الـ dispatcher يمنع الـ Push/Email خلال هذه الساعات إلا لو الـ priority = 'URGENT'.

# Wave 06 — Architecture Decision Records

## ADR-0086: Block Editor Architecture — No TipTap in Production Bundle

**Status:** Accepted  
**Date:** 2026-05-17  
**Deciders:** Zenith Engineering

### Context
Wave 06 requires a rich block editor. TipTap is the industry standard but adds ~200KB to the bundle.

### Decision
Use native `contenteditable` with custom block renderer for initial release. TipTap can be added in Wave 08 when formula blocks require advanced extension support.

### Consequences
- ✅ Bundle size < 100KB for editor
- ✅ No TipTap dependency conflicts
- ⚠️ Less rich text formatting in v1 (acceptable per roadmap)

## ADR-0087: Fractional Indexing for Block Order

**Status:** Accepted  
**Date:** 2026-05-17

### Context
Block reordering needs an efficient positional system without full re-indexing.

### Decision
Use floating-point fractional indexing (`position DOUBLE PRECISION`). Renormalize via `reorder_block` RPC when gap < 1e-9.

### Consequences
- ✅ O(1) insert/reorder
- ✅ No collisions for ≥ 10^9 operations
- ⚠️ Renormalize needed after extreme fragmentation (automatic via trigger)

## ADR-0088: Idempotency-Key on Every Block Mutation

**Status:** Accepted  
**Date:** 2026-05-17

### Context
Network failures can cause duplicate mutations. Users expect exactly-once semantics.

### Decision
Every mutation (create/update/delete/move) requires an `Idempotency-Key` header (min 16 chars). DB has `UNIQUE` constraint on `(page_id, idempotency_key)`.

### Consequences
- ✅ Safe retry after network failure
- ✅ Outbox pattern works correctly
- ⚠️ Key must be generated client-side (via `newIdempotencyKey()`)

## ADR-0089: Vault Block ≠ AI Context

**Status:** Accepted  
**Date:** 2026-05-17

### Context
Vault blocks contain sensitive user data. AI context must never include vault content.

### Decision
`assertNoVaultBlockInAIContext(blocks)` throws if any vault block is passed to AI. This is enforced in both service layer and DB trigger.

### Consequences
- ✅ Zero-knowledge vault protection
- ✅ CI scan prevents accidental vault→AI leaks
- ⚠️ Developer must explicitly filter vault before calling AI

## ADR-0090: Outbox Queue via localStorage (No Server Required)

**Status:** Accepted  
**Date:** 2026-05-17

### Context
Offline editing requires persisting mutations when offline.

### Decision
Failed mutations are queued in `localStorage['zenith:block:outbox']` (max 200 entries). On reconnect, `ConnectivityIndicator` flushes the queue. Secrets are never stored in outbox.

### Consequences
- ✅ Works without service worker
- ✅ Secrets-safe (content sanitized before queue)
- ⚠️ Tab-local only (not cross-tab sync in v1)

## ADR-0091: Cycle Detection in Synced Blocks

**Status:** Accepted  
**Date:** 2026-05-17

### Context
Synced blocks can create circular references leading to infinite render loops.

### Decision
Before inserting a `synced_block`, `detectCycle()` runs on the full block graph. Also enforced via DB trigger.

### Consequences
- ✅ No infinite loops
- ✅ O(V+E) detection per insert
- ⚠️ Slight latency on synced_block creation (acceptable)

## ADR-0092: Virtual Scroll for 200+ Blocks

**Status:** Accepted  
**Date:** 2026-05-17

### Context
Pages with 200+ blocks cause DOM performance issues.

### Decision
`VirtualBlocks` component uses native `scroll` event + estimated row height (48px). Fallback to full render for ≤50 blocks. No external virtualization library to minimize bundle.

### Consequences
- ✅ 60fps scroll with 1000+ blocks
- ✅ Zero extra dependencies
- ⚠️ Variable row heights cause slight position drift (acceptable in v1)

## ADR-0093: RTL-First Block Editor

**Status:** Accepted  
**Date:** 2026-05-17

### Context
Primary users write in Arabic (RTL). Editor must feel native.

### Decision
- `dir="auto"` on all text blocks
- `dir="ltr"` hardcoded on code blocks
- `insetInlineStart`/`End` used instead of `left`/`right` throughout
- Tested with both Arabic and English content

### Consequences
- ✅ Seamless bidi text
- ✅ WCAG 2.2 AA for RTL
- ⚠️ Mixed-direction content may need manual override

## ADR-0094: Files Upload via Signed URLs Only

**Status:** Accepted  
**Date:** 2026-05-17

### Context
Files must not pass through the application server (performance + security).

### Decision
`request_file_upload` RPC returns `{ file_id, object_key, bucket }`. Client uses Supabase Storage SDK to get signed URL and uploads directly to bucket. MIME whitelist enforced at DB level.

### Consequences
- ✅ Files never touch app server
- ✅ MIME check at DB level
- ✅ 5GB quota per workspace (equal, no paid tiers)
- ⚠️ Virus scan runs async via worker (not blocking upload)

## ADR-0095: DOMPurify for All Block Content

**Status:** Accepted  
**Date:** 2026-05-17

### Context
Block content_json may contain HTML from markdown import or user paste.

### Decision
`sanitizeBlockContent()` runs DOMPurify on all string values before DB insert. Allowed tags: `b, i, u, s, code, a, br`. No `script`, `style`, `iframe` in content.

### Consequences
- ✅ XSS-safe block content
- ✅ Covers all 25+ block types
- ⚠️ Rich HTML stripped to allowed subset (expected behavior)
